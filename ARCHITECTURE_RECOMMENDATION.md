# Рекомендации по архитектуре проекта

## Сайт для пользователей (клиентская часть)

### Рекомендация: **Отдельный проект**

Создайте отдельный проект для клиентского сайта. Это правильный подход по следующим причинам:

### Преимущества отдельного проекта:

1. **Разделение ответственности**
   - Админ-панель и клиентский сайт имеют разные цели и аудиторию
   - Разные требования к безопасности и функциональности
   - Разные дизайны и UX

2. **Независимое развертывание**
   - Можно обновлять админ-панель и клиентский сайт независимо
   - Разные домены/поддомены (admin.restaurant.com и restaurant.com)
   - Разные CDN и оптимизации

3. **Безопасность**
   - Админ-панель требует более строгой аутентификации
   - Клиентский сайт может быть публичным с ограниченным доступом
   - Разные политики CORS и безопасности

4. **Производительность**
   - Клиентский сайт может быть оптимизирован для SEO и быстрой загрузки
   - Админ-панель может иметь больше зависимостей и функций
   - Разные стратегии кэширования

5. **Масштабируемость**
   - Легче масштабировать каждое приложение отдельно
   - Разные команды могут работать над разными проектами
   - Проще тестировать и поддерживать

### Что нужно для клиентского сайта:

1. **Страницы:**
   - Главная страница (меню, информация о ресторане)
   - Страница меню с фильтрами и поиском
   - Страница бронирования столика
   - Личный кабинет (история заказов, бронирования)
   - Регистрация/Вход

2. **Функциональность:**
   - Просмотр меню с категориями
   - Бронирование столика (выбор даты, времени, количества гостей)
   - Просмотр истории заказов (если пользователь зарегистрирован)
   - Профиль пользователя

3. **API интеграция:**
   - Использовать тот же backend API
   - Эндпоинты для клиентов:
     - `POST /api/auth/signup` - регистрация
     - `POST /api/auth/signin` - вход
     - `GET /api/dishes` - получение меню
     - `POST /api/reservations` - создание бронирования
     - `GET /api/reservations/client/{id}` - бронирования клиента
     - `GET /api/orders/client/{id}` - заказы клиента

### Технологии для клиентского сайта:

**Рекомендуемый стек:**
- **React + TypeScript** (как в админ-панели) - для единообразия
- **Vite** - для быстрой сборки
- **Ant Design** или **Material-UI** - для UI компонентов
- **React Router** - для навигации
- **Axios** - для API запросов
- **Zustand** - для управления состоянием

**Альтернативы:**
- **Next.js** - если нужен SSR и SEO
- **Vue.js** - если команда предпочитает Vue
- **SvelteKit** - для максимальной производительности

### Быстрый старт нового проекта:

```bash
# Создать новый проект
npm create vite@latest restaurant-client -- --template react-ts

# Установить зависимости
cd restaurant-client
npm install antd axios react-router-dom zustand dayjs

# Настроить структуру
mkdir -p src/pages src/components src/api src/store src/types
```

### Общие компоненты (опционально):

Если хотите переиспользовать код, можно создать:
- **Монорепозиторий** с общими пакетами (shared API client, types, utils)
- **NPM пакеты** для общих компонентов
- **Git submodules** для общих модулей

Но для начала лучше просто скопировать и адаптировать:
- API клиент (`src/api/client.ts`)
- Типы (`src/types/api.ts`)
- Утилиты (`src/utils/`)

### Итоговая рекомендация:

✅ **Создайте отдельный проект** `restaurant-client` для клиентского сайта
✅ Используйте тот же backend API
✅ Используйте похожий стек технологий для единообразия
✅ Разделите развертывание и домены

Это стандартная практика в индустрии и обеспечит лучшую архитектуру проекта.
